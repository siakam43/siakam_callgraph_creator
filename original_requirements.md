# 原始需求


## 整体说明
1. 我需要开发一个单纯由markdown文档和python脚本组成的skill, 名字叫siakam_callgraph_creator
2. 核心功能：生成c语言项目的call graph，要求高准确率

## 流程与架构

### 0 工具输入
1. SKILL使用方法：/siakam_callgraph_creator project_dir
2. 参数project_dir为待分析项目路径，如果用户没有指定该参数，默认为当前文件夹
3. 在project_dir中，如果存在.siakamignore文件，需要将该文件中指定的子目录或文件排除掉，不进行分析。.siakamignore文件语法和.gitignore语法一样。

### 1 模块A
1. 模块A的功能是：a 解析.siakamignore文件，确定分析范围   b 识别出项目中的全部函数，即call graph中的点，具体信息内容待设计（函数名，路径，是否有代码实现等信息）。 c 识别出全部直接函数调用，即call graph中的边，具体信息内容待设计（caller, callee, 代码位置等）。  d 针对每个有函数体的函数，识别出该函数中存在的全部间接调用点（函数指针）。注意这里只需识别出有间接调用即可，无需分析函数指针的具体目标。具体信息内容待设计（位置，函数指针名字或表达式等）
2. 模块A的功能全部由python实现, 代码解析建议使用tree_sitter。
3. 工具不要求代码可编译，工具分析不能依赖编译。可以要求代码语法正确。遇到语法错误的代码，给出提示，跳过错误部分，不要影响其它部分的分析。
4. 该模块在被分析代码语法正确的前提下，要求准确率100%，0漏报，0误报。该模块python工具开发时，需要配套功能测试。
5. 该模块输出文件保存在project_dir中的.siakam_out文件夹中。全部识别的结果信息保存在json文件中。格式待设计。

### 2 模块B
1. 模块B的功能是：深入分析模块A识别找到的全部间接调用，明确间接调用的caller-callee对应关系
2. 模块B的功能由LLM完成分析。本模块由markdown文档组成，在Agent(claude/opencode)调用skill时，会使用LLM完成分析任务。重要！本模块禁止使用任何脚本和代码！
3. 本模块分析任务多，但相对独立。提示词应该要求采取多subagent并行分析的策略。
4. 需要考虑任务异常中断的场景，每个间接调用完成分析后，需要将结果单独保存在一个文件中，输出路径为project_dir中的.siakam_out文件夹。每个间接调用输出的结果文件名（待设计）需要和该间接调用直接相关，当异常中断后，重新分析时，已有结果的间接调用不要重复分析。
5. 间接调用分析任务难度大，需要使用代码仓探索的常用工具，SKILL应提前申请权限。
6. LLM应尽最大努力，精准分析出间接调用的详情。


### 3 模块C
1. 模块C的功能是：综合模块A和模块B的结果，生成完整的callgraph及其它结果文件。输出路径是project_dir中的.siakam_out文件夹
2. 输出a callgraph.json 和 callgraph.dot，两种格式的callgraph，包含直接调用和间接调用
3. 输出b indirect_call.json, 汇总模块B识别的全部间接调用信息
4. 输出c entry.json，分析callgraph，找到全部有函数体（代码实现），但没有被其它函数调用的函数（入度为0的点），整理成列表输出。这部分功能可以使用Python脚本实现。

## 要求
1. 工具全流程需要考虑异常中断场景，每一步结果都要存入文件，且下次任务能够精准找到。建议为每个间接调用建立一个uid（间接调用的信息计算一个hash）。模块A识别的间接调用结果中，用uid标记每个间接调用。模块B在分析每个间接调用后，将结果保存在以uid命名的结果文件中。任务中断重启后，已经有结果文件的间接调用将不再分析。
2. 模块B是工具的核心，难度大，由LLM review代码的方式完成，SKILL提示词是本工具开发的重点。需要标准的提示词，创建任务列表等方式严格限制LLM的行为，避免发生任务遗忘。同时，要注意不同间接调用分析过程的相互影响。建议本模块也需要skill的配套功能测试。
